# Proposal

## Why

最近一批桌面端、Skill、Plugin、MCP、同步和发布相关改动暴露出一个共同问题：代码可能已经提交，但验证记录没有稳定证明功能已经能被真实用户直接使用。

本变更把 PromptHub 的验收标准从“跑过部分命令”提升为“功能可用、UI 实操通过、复用边界清楚、逻辑分支被测试、静态审计有记录、防护路径有覆盖”。目标是让后续 agent 和贡献者在提交前能按同一套门槛自查，而不是只依赖聊天上下文或主观判断。

## Scope

- In scope:
  - 补充 UI 实操验证和截图/浏览器自动化要求。
  - 补充直接可用、无明显 bug 的验收基线。
  - 补充 UI/逻辑复用审计要求，避免重复造轮子。
  - 补充单元测试、白盒分支、静态扫描、安全防护和性能压力的记录要求。
  - 同步项目级 verification、testing standards、Definition of Done 和 code quality rules。
- Out of scope:
  - 本轮不直接补具体业务模块的单元测试或 E2E 用例。
  - 本轮不改发布脚本、CI 工作流或应用运行逻辑。
  - 本轮不关闭或归档历史变更。

## Risks

- 标准变严后，短期内更多变更会暴露验证缺口。
- 如果只写规则但不在后续实施中执行，文档会再次变成形式化清单。
- UI 实操验证需要运行桌面端或浏览器页面，可能受本机依赖、签名、网络或 Electron 环境影响。

## Rollback Thinking

本变更只更新 spec 和规则文档。若发现标准过重，可以在同一 active change 中调整验收矩阵的适用范围，而不影响生产代码。
